# HackSummit Builder Challenge 2026 — Track 2

**Event:** April 22–23, 2026, Lausanne, Beaulieu Halle 36
**Track:** Track 2 — Product authenticity verification via smartphone scanning of SICPA security codes
**Sponsor:** SICPA (provides code samples and technical constraints on the day — NO live API)

## Challenge Summary
Design a simple, intuitive smartphone experience for verifying product authenticity
by scanning a SICPA security code. Track framed as UX/UI design, but the real
constraint is SICPA's CDP scanning technology.

## Current Status
**App complete and demo-ready.** All 5 screens + gamification layer built and running.
Tested on iPhone 13 mini. Next: test on physical device at the venue, prepare pitch deck.

## The Strategy (read track2_strategy.txt for full detail)

**Angle:** "You built the vault. We built the bank."
SICPA has a working backend but no universal consumer-grade front door. Their
existing apps (DGDA Connect, GRS-SICPA Connect, CalOrigin, Hakiki) are
government-mandated white labels — nobody installs them voluntarily.

**Hackathon deliverable:** One user journey. Young adult, scanning a product
in a pharmacy/grocery store. 5 screens. The UX innovation is in screen 2
(guided capture): haptic feedback + visual frame guide + auto-trigger.
Every scan quietly contributes to a store trust score (seeds the network idea).

**Pitch structure:**
- Min 1: The gap in SICPA's existing apps
- Min 2: Live demo of the 5-screen flow
- Min 3: Trust aggregation network as roadmap ("the 1% protect the 99%")
- Close: "You built the vault. We built the bank."

## The 5 Screens (all built)
1. Home — single "Scan a Product" button, no onboarding, no tab bar
2. Guided Capture — the key innovation: haptic ticks + visual frame + auto-trigger
3. Processing — "Reading security signature..." (mocked API)
4. Result — AUTHENTIC/SUSPICIOUS/COUNTERFEIT + product details + XP pop animation
5. Contribution — store trust score + quest badge toasts (slide in from top)

Tab bar (Scan / Me) is visible only on screens 4–5 and the Profile screen.
Home, Capture, and Processing have no tab bar — the UI stays clean until the scan completes.

## Mocked API Contract
Input:  { image: base64, location: { lat, lng }, product_hint: string }
Output: { status: "authentic"|"suspicious"|"fake", confidence: 0.0-1.0,
          product: { name, brand, batch, manufactured_at },
          scan_count_here_today: int }

## Gamification
Implemented. Key design rules (all logic in src/store/xpStore.js):
- XP per scan: authentic=1, suspicious=2, fake=3 (higher reward for catching fakes)
- 10 levels: Civilian (0) → Scout → Tracker → Sentinel → Warden → Investigator →
  Inspector → Guardian → Argus Elite → Argus Master (500+)
- Streaks are non-punishing: missing a period resets the run to 1 on next scan,
  not to 0. Only the most significant active streak is shown (daily has priority).
- XP pill overlay visible on all screens except Home and Capture.

## Key Facts About SICPA (from pre-event research)
- CDP = Copy Detection Pattern. Micro-texture encoding. Requires precise capture.
- Existing apps are B2G/B2B white labels. No universal consumer app exists.
- SICPATRACE® Evo = their backend platform.
- Digital Trust Suite = their newer digital identity layer (W3C VCs, DIDs).
- No API available on hackathon day — participants mock the response.
- SICPA explicitly welcomes NFC-based or non-SICPA-code alternatives ("go crazy").
- Deep learning attacks on CDP are a known threat — trust aggregation is a defense.

## Key Design Decisions
- Elderly users: do NOT design a direct scan flow for them. They benefit passively
  via the store trust score generated by younger users (generational trust relay).
- Gamification is part of the core flow (XP, levels, badges, quests on Me tab).
- No account creation required to start scanning.
- CDP complexity is completely hidden from the user by the guided capture UX.

## CaptureScreen calibration (non-obvious constants)
The distance thresholds in CaptureScreen.js are physics-derived for iPhone 13 mini:
areaFrac(D) = 4.884 / D² (D in cm, portrait mode). Key values:
- IDEAL_MIN = 0.02 (≈ 15.6 cm) — closest safe focus distance
- IDEAL_MAX = 0.035 (≈ 11.8 cm, at focus limit)
- ZOOM_START_FRAC = 0.014 — auto-zoom activates here (≈ 18.7 cm out)
- ZOOM_MAX = 0.1 (1.5× zoom)
Before changing these, re-read the derivation comment block at the top of CaptureScreen.js.

## Contingency
If CDP scanning is too unreliable to demo on the day: drop guided capture,
replace with standard QR scan, shift narrative to trust aggregation network.
